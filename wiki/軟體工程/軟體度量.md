# 軟體度量 (Software Metrics)

**標籤**: #軟體工程 #度量 #品質管理 #圈複雜度 #功能點 #技術債 #DevOps
**日期**: 2024-01-15

軟體度量是用數量化方式評估軟體產品、開發過程與資源的指標，幫助團隊做出數據驅動的決策。透過系統性的度量，組織可以追蹤進度、發現問題、預測風險並持續改進軟體品質。軟體度量學起源於 1970 年代，當時軟體危機促使工程師尋求更科學的管理方法。

## 軟體度量的發展歷程

### 早期度量 (1970s-1980s)

最早的軟體度量主要關注程式碼規模，如程式碼行數 (LOC)。這個時期的度量簡單直接，但缺乏理論基礎。

### 過程度量興起 (1990s)

隨著軟體工程成熟，人們開始關注開發過程的度量，如需求變更率、缺陷密度等。CMM（能力成熟度模型）強調度量的重要性。

### 現代度量 (2000s-現在)

敏捷開發和 DevOps 推動了新的度量範式，如部署頻率、交付週期、變更前置時間等 DORA 度量，以及 DevOps 能力成熟度指標。

## 產品度量

### 規模度量

**程式碼行數 (Lines of Code, LOC)** 是最直觀的規模指標。分為原始碼行數 (SLOC)、邏輯行數和總行數。LOC 的優點是簡單易懂，但缺點是受程式語言、編碼風格影響，且不能反映程式複雜度或品質。使用時應注意：不要用於跨語言比較、不要作為團隊績效考核的唯一指標、要關注趨勢而非絕對值。

**功能點 (Function Points, FP)** 是基於軟體功能的規模度量，由 Alan Albrecht 在 1979 年提出。功能點計算基於使用者功能需求：外部輸入、輸出、查詢、內部邏輯檔案、外部介面檔案。調整後功能點 (AFP) = ΣFP × 調整因子。優點是與程式語言無關、適合估算和規劃，但計算複雜且主觀性較強。

**物件導向度量** 針對 OO 語言設計：類別數 (noc)、類別平均方法數 (nom)、繼承深度 (dit)、物件間耦合度 (noc)、回響集合 (rfc)。這些指標有助於評估設計品質。

### 品質度量

**缺陷密度 (Defect Density)** = 缺陷數 / 程式碼規模（常用千行為單位）。業界標準參考：優秀 < 1/KLOC，良好 1-2/KLOC，普通 2-5/KLOC，差 > 5/KLOC。但此指標不應作為品質唯一標準，因為測試深度會影響發現的缺陷數。

**測試覆蓋率** 分為程式碼覆蓋率和需求覆蓋率。語句覆蓋率表示執行的語句數/總語句數。分支覆蓋率表示執行的分支數/總分支數。路徑覆蓋率表示執行的路徑數/總路徑數。高覆蓋率不等於高品質，但低覆蓋率通常表示高風險。

**平均故障間隔 (MTBF)** 衡量系統可靠性：MTBF = 總運行時間 / 故障次數。相關指標還有平均修復時間 (MTTR) 和可用性 (Availability) = MTBF / (MTBF + MTTR)。

## 過程度量

### 傳統開發度量

交付速度衡量每單位時間交付的功能數量。目標是穩定成長而非追求峰值。需求變更率 = 需求變更次數 / 總需求數。目標應低於 20%，過高表示需求不穩定或分析不足。累計技術債記錄修復技術債所需時間。目標是持續減少而非增加。

### DevOps 度量

**DORA 四大指標** 是 Google 提出的 DevOps 能力衡量標準。部署頻率 (Deployment Frequency)：從 daily 到 per hour，反映交付能力。變更前置時間 (Lead Time for Changes)：從 commit 到部署的時間，反映流程效率。平均恢復時間 (Mean Time to Recovery, MTTR)：故障發生到恢復的時間，反映 reliability。變更失敗率 (Change Failure Rate)：部署失敗的比例，反映穩定性。

| 等級 | 部署頻率 | 變更前置時間 | MTTR | 變更失敗率 |
|------|----------|--------------|------|------------|
| Elite | 按需 | < 1 小時 | < 1 小時 | 0-15% |
| High | 每週-每月 | 1 天-1 週 | < 1 天 | 16-30% |
| Medium | 每月-半年 | 1-6 個月 | 1 天-1 週 | 16-30% |
| Low | 半年以上 | > 6 個月 | > 6 個月 | > 30% |

### 敏捷度量

衝刺速度 (Sprint Velocity) 衡量每個衝刺完成的點數。燃盡圖 (Burn-down Chart) 顯示剩餘工作隨時間的變化。累積流量圖 (Cumulative Flow Diagram) 顯示各狀態工作項的累積變化。這些度量幫助團隊預測進度和發現問題。

## 圈複雜度 (Cyclomatic Complexity)

圈複雜度是 Thomas McCabe 於 1976 年提出的程式碼複雜度指標，衡量控制流分支的數量。

### 數學定義

對於流程圖，CC = E - N + 2P，其中 E 是邊數，N 是節點數，P 是連通分量數。對於單一函數這簡化為 CC = 2 - N，其中 N 是分支節點數（if、while、for 等）數量。

### 等級對照

| CC 值 | 等級 | 風險 | 建議 |
|-------|------|------|------|
| 1-10 | 低 | 簡單，無風險 | 無需特殊處理 |
| 11-20 | 中 | 中等複雜度 | 可接受，需關注 |
| 21-50 | 高 | 複雜，測試困難 | 重構建議 |
| 51-75 | 極高 | 難以維護 | 必須重構 |
| >75 | 無法控制 | 極高風險 | 立即重構 |

### Python 計算範例

```python
import ast

def calculate_cyclomatic_complexity(code):
    """計算函式的圈複雜度"""
    tree = ast.parse(code)
    complexity = 1  # 基礎複雜度為 1
    
    for node in ast.walk(tree):
        if isinstance(node, (ast.If, ast.While, ast.For, ast.ExceptHandler)):
            complexity += 1
        elif isinstance(node, ast.BoolOp):
            # and/or 增加複雜度
            complexity += len(node.values) - 1
    
    return complexity

# 測試
test_code = """
def process(x):
    result = 0
    if x > 0:
        if x < 10:
            result = x * 2
        else:
            result = x + 1
    else:
        result = 0
    return result
"""

print(f"圈複雜度: {calculate_cyclomatic_complexity(test_code)}")  # 3
```

### 降低複雜度的方法

提取方法將複雜邏輯拆分為多個小函式。替換條件式使用策略模式或多型。簡化布林表達式使用德摩根定律。減少巢狀結構使用早期返回。

## 技術債度量

### 技術債的類型

技術債可分為多種：程式碼級技術債包括壞味道程式碼、重複程式碼、缺乏測試。架構級技術债包括耦合過高、缺乏抽象、決策錯誤。文件級技術债包括缺少文件、過時文件。測試級技術债包括低覆蓋率、脆弱測試。

### 度量方法

**SonarQube** 提供技術債比率 (Technical Debt Ratio) = 修復時間 / 編碼時間。目標應低於 5%。它還提供清除技術債所需時間的估算。**CodeClimate** 提供 GPA (0-4) 和技術債美元價值估算。

```python
# SonarQube API 範例
import requests

def get_technical_debt(server_url, project_key, token):
    url = f"{server_url}/api/measures/component"
    params = {
        'component': project_key,
        'metrics': 'sqale_debt_ratio,new_technical_debt_ratio',
        'branch': 'main'
    }
    headers = {'Authorization': f'Bearer {token}'}
    
    response = requests.get(url, params=params, headers=headers)
    return response.json()
```

## 實踐建議

### 選擇有意義的度量

度量應符合團隊目標和組織戰略。不同階段關注不同指標：啟動階段關注需求穩定性和估算準確性。開發階段關注程式碼品質和技術債。測試階段關注缺陷密度和覆蓋率。運維階段關注 MTTR 和可用性。

### 避免遊戲化

單一指標容易被人為操作，可能造成負面影響。例如追求 LOC 可能導致程式碼冗長。追求測試覆蓋率可能添加無意義測試。追求部署頻率可能忽略穩定性。解決方法是使用多個互補指標，並結合定性分析。

### 趨勢分析

單一數值意義有限，追蹤變化趨勢更有價值。設定基準線並持續監控。識別異常並探究原因。比較不同團隊或專案時要謹慎。

### 結合上下文

度量需要配合其他分析方法。程式碼審查提供品質的定性評估。技術分享促進知識傳遞。 retrospectives 反思過程改進。客戶回饋了解實際價值。

## 常用工具

### 靜態分析工具

**SonarQube** 是開源的程式碼品質管理平臺，支持 20+ 語言，提供持續程式碼品質檢查，追踪技術债，支持 Quality Gate。

**CodeClimate** 是雲端程式碼分析平臺，易於集成 GitHub/GitLab，提供 GPA 評分，自動檢測常見問題。

**ESLint/TSLint** 是 JavaScript/TypeScript 的 linter，可自定義規則，與 CI/CD 集成。

### 程式碼行數工具

**SLOCcount** 支持多語言的程式碼統計。**cloc** 可以分析程式碼行數並忽略註解和空白。**tokei** 是快速的 Rust 實現。

### CI/CD 集成

**Jenkins Metrics Plugin** 收集 Jenkins 構建指標。**GitLab Metrics** 提供 DevOps 度量儀表板。**GitHub Insights** 追踪開發活動。

## 建立度量文化的建議

### 從小事做起

選擇團隊最關心的 2-3 個指標開始。確保數據準確和可收集。定期回顧並調整。

### 教育與培訓

讓團隊理解度量的意義。避免將度量用於懲罰。建立心理安全的環境。

### 持續改進

度量只是手段，不是目的。定期評估度量的有效性。根據學習調整策略。
