# 版本控制

版本控制系統是追蹤檔案變化歷史、支援多人協作的核心工具。它記錄檔案或專案的每次修改，允許開發者回溯到任何歷史版本、比較不同版本的差異、合併多人同時編輯的結果。在現代軟體開發中，版本控制不僅是程式碼管理的基礎，更是團隊協作、持續整合、部署自動化的關鍵環節。從個人專案到大型企業級開發，版本控制系統已經成為軟體工程不可或缺的工具。

## 版本控制的重要性

### 核心功能

```
┌─────────────────────────────────────────────────────────────────┐
│                      版本控制功能                               │
├─────────────────────────────────────────────────────────────────┤
│                                                              │
│  • 歷史追蹤：記錄每次修改的時間、作者、內容                     │
│  • 版本回溯：回到任意歷史版本                                   │
│  • 差異比較：視覺化展示版本之間的變化                           │
│  • 分支管理：並行開發不同功能                                   │
│  • 合併解決：自動或手動解決合併衝突                             │
│  • 備份还原：異地備份，防止資料遺失                             │
│  • 權限控制：管理誰可以修改什麼                                 │
│                                                              │
└─────────────────────────────────────────────────────────────────┘
```

### 團隊協作價值

```
協作效益：
• 獨立開發：每人可在自己的分支工作
• 程式碼審查：透過 Pull Request 審查程式碼
• 責任追溯：知道是誰在何時做了什麼修改
• 衝突預警：及早發現並解決衝突
• 知識共享：透過歷史記錄學習團隊程式碼
```

## 版本控制系統類型

### 集中式 vs 分散式

| 特性 | 集中式 (SVN, CVS) | 分散式 (Git, Mercurial) |
|------|-------------------|-------------------------|
| 儲存庫 | 中央伺服器單一儲存庫 | 每個開發者完整儲存庫 |
| 離線工作 | 需要連線 | 完全離線 |
| 分支 | 較慢，可能衝突 | 快速、輕量 |
| 合併 | 複雜 | 強大工具支援 |
| 依賴 | 伺服器可用性 | 無中央依賴 |

### 主流系統比較

```python
# Git 特性
# • 最流行的分散式 VCS
# • 功能強大，靈活度高
# • 龐大生態系 (GitHub, GitLab)
# • 學習曲線較陡

# Mercurial 特性
# • 類似 Git 但更易上手
# • 效能優異
# • 較小社群

# Subversion (SVN) 特性
# • 集中式，易理解
# • 適合大型單一儲存庫
# • 逐步被 Git 取代

# Perforce 特性
# • 企業級，大型專案
# • 高效處理大量檔案
# • 商業軟體
```

## Git 基本指令

### 初始化與配置

```bash
# 初始化新儲存庫
git init

# 克隆現有儲存庫
git clone https://github.com/user/repo.git

# 設定使用者資訊
git config --global user.name "Your Name"
git config --global user.email "your@email.com"

# 檢視設定
git config --list
git config --global --list
```

### 基本操作

```bash
# 檢視狀態
git status

# 檢視差異
git diff              # 工作區 vs 暫存區
git diff --staged     # 暫存區 vs 最近提交
git diff HEAD        # 工作區 vs 最近提交
git diff commit1 commit2  # 兩次提交差異

# 階段檔案
git add filename         # 單一檔案
git add .                # 所有變更
git add -p               # 部分階段

# 提交
git commit -m "提交訊息"
git commit --amend      # 修改最後提交
git commit -a -m "訊息" # 自動追蹤已修改檔案
```

### 提交歷史

```bash
# 檢視歷史
git log
git log --oneline        # 精簡格式
git log -n 10            # 最近 10 筆
git log --graph         # 圖形化
git log --author="name" # 按作者過濾
git log --grep="keyword" # 按訊息關鍵字
git log -p filename     # 檔案變更歷史

# 檢視特定提交
git show commit_id
git show HEAD~3         # 前 3 次提交
```

### 分支操作

```bash
# 分支管理
git branch                 # 列出分支
git branch -a             # 所有分支
git branch -v             # 詳細資訊
git branch new-branch     # 建立分支
git branch -d branch-name # 刪除分支
git branch -m old new     # 重新命名

# 切換分支
git checkout branch-name
git checkout -b new-branch  # 建立並切換
git switch branch-name      # 建議用法
git switch -c new-branch    # 建立並切換

# 合併分支
git merge branch-name      # 合併到目前分支
git merge --no-ff branch   # 禁止快轉合併
git rebase main            # 變基到 main
```

### 遠端操作

```bash
# 遠端管理
git remote -v              # 檢視遠端
git remote add origin url  # 新增遠端
git remote remove origin  # 移除遠端
git remote set-url origin new-url # 修改 URL

# 推送
git push origin main       # 推送到 main
git push -u origin feature # 設定上游並推送
git push --force           # 強制推送（小心！）
git push --tags           # 推送標籤

# 拉取
git pull origin main       # 拉取並合併
git fetch origin          # 只抓取不合併
git pull --rebase        # 用變基代替合併
```

### 標籤操作

```bash
# 建立標籤
git tag v1.0.0            # 輕量標籤
git tag -a v1.0.0 -m "訊息" # 註釋標籤

# 推送標籤
git push origin v1.0.0
git push --tags          # 推送所有標籤

# 檢視標籤
git tag
git show v1.0.0
```

## 分支策略

### Git Flow

適合有正式發布週期的專案：

```
┌─────────────────────────────────────────────────────────────────┐
│                      Git Flow                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                              │
│  main ───●────●────●────●────●───                        │
│           ↑     ↑                                     │
│        release                                    │
│           ↓     ↓                                     │
│  develop ─●────●────●────●────●────●───              │
│                ↑          ↑                      │
│             feature    feature                    │
│                                                              │
│  分支說明：                                                 │
│  • main: 生產環境程式碼                                     │
│  • develop: 開發主分支                                     │
│  • feature: 功能分支 (develop 衍生)                         │
│  • release: 發布分支 (develop 衍生)                         │
│  • hotfix: 緊急修復 (main 衍生)                              │
│                                                              │
└─────────────────────────────────────────────────────────────────┘
```

```bash
# Git Flow 命令
git flow init
git flow feature start my-feature
git flow feature finish my-feature
git flow release start v1.0
git flow release finish v1.0
git flow hotfix start v1.0.1
git flow hotfix finish v1.0.1
```

### GitHub Flow

適合持續部署的專案：

```bash
# GitHub Flow 流程
# 1. 從 main 建立分支
git checkout -b feature/my-feature

# 2. 開發並提交
git add .
git commit -m "Add feature"

# 3. 開啟 Pull Request
# 在 GitHub/GitLab 介面操作

# 4. 討論和審查

# 5. 合併到 main
# 透過介面合併或
git checkout main
git merge feature/my-feature
git push origin main

# 6. 部署
```

### Trunk-Based Development

適合小型、快速迭代的團隊：

```python
# 特色：
# • 直接在 main/trunk 開發
# • 小功能尽快合併
# • 需要高自動化測試
# • CI/CD 必須完善

# 實踐：
# • Feature Flag 控制功能上線
# • 短生命周期分支 (< 2 天)
# • 高測試覆蓋率
# • 快速回饋迴圈
```

## 進階技巧

### Stash

```bash
# 儲存未提交的變更
git stash
git stash save "work in progress"

# 檢視儲存
git stash list

# 恢復儲存
git stash pop        # 恢復並刪除
git stash apply      # 恢復但保留
git stash drop       # 刪除儲存
```

### 交互式變基

```bash
# 壓縮提交
git rebase -i HEAD~3

# 交互式變基
# pick -> reword: 修改訊息
# pick -> edit: 暂停修改
# pick -> squash: 與前一個合併
# pick -> drop: 刪除提交
```

### 子模組

```bash
# 新增子模組
git submodule add https://github.com/user/repo path/to/submodule

# 初始化子模組
git submodule init

# 更新子模組
git submodule update
git submodule update --recursive
```

### 子樹

```bash
# 新增子樹
git subtree add --prefix=lib https://github.com/user/lib.git main --squash

# 更新子樹
git subtree pull --prefix=lib https://github.com/user/lib.git main --squash
```

## 程式碼審查

### Pull Request 工作流

```bash
# 建立 Pull Request
# 1. 推送分支
git push -u origin feature/my-feature

# 2. 在 GitHub/GitLab 開啟 PR

# 3. 描述變更內容
# - 解決什麼問題
# - 實作方式
# - 測試結果

# 4. 請求審查

# 審查流程
# - 檢視程式碼
# - 提供回饋
# - 討論問題
# - 批准/要求修改
```

### 審查要點

```
程式碼審查重點：
• 正確性：邏輯是否正確
• 可讀性：命名、註解、结构
• 效能：是否有優化空間
• 安全性：是否有漏洞
• 測試：是否有充分測試
• 風格：是否符合專案規範
```

## 最佳實踐

### 提交訊息規範

```bash
# 好的提交訊息
# 標題：簡短說明（< 50 字）

# 正文：
# - 動機：為什麼需要這個變更
# - 變更：具體改了什麼
# - 影響：可能的副作用

# 範例：
# Add user authentication module
#
# Implement JWT-based authentication for API endpoints.
# - Add login/logout endpoints
# - Add token validation middleware
# - Add password hashing
#
# May affect performance for large user databases.
```

### 避免的行為

```python
# 不要做：
# • 提交編譯不過的程式碼
# • 提交未測試的功能
# • 提交包含敏感資訊的檔案
# • 強制推送到主分支
# • 提交大型二進位檔案（應使用 Git LFS）
# • 在提交訊息中留下 TODO
```
