# 容器編排 (Container Orchestration)

## 概述

容器編排（Container Orchestration）是自動化管理、部署、擴展和網路連接多個容器的技術，是現代雲端原生應用的核心基礎設施。

## 為何需要容器編排

```
手動管理                          編排管理
─────────                        ─────────
容器數量少時可行                  數十到數千個容器
手動部署簡單                      需要自動化部署
故障時手動重啟                    自動健康檢查和重啟
手動擴展                         自動和手動擴展
單一主機                        多主機叢集
```

### 核心資源物件

#### Pod
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
    tier: frontend
spec:
  containers:
  - name: nginx
    image: nginx:1.21
    ports:
    - containerPort: 80
    resources:
      limits:
        memory: "128Mi"
        cpu: "500m"
      requests:
        memory: "64Mi"
        cpu: "250m"
    livenessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 3
      periodSeconds: 10
```

#### Deployment
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
```

#### Service
```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  type: ClusterIP
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
```

## 部署策略

### Rolling Update
```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
```
逐步替換舊版 Pod，零停機。

### Blue-Green Deployment
```
v1 (Blue) ──────────────────→ 流量 100%
v2 (Green) ─建立完成──→ 切換流量 ──→ v2 接管
                        v1 保持備用
```

### Canary Deployment
```
v1: ████████████████████ 90%
v2: ████ 10% (測試流量)
```

## 自動擴展 (Auto Scaling)

### Horizontal Pod Autoscaler (HPA)
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx-deployment
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

### 自動擴展流程
```
┌──────────┐    CPU > 70%    ┌──────────┐
│ Prometheus│ ────────────→  │  HPA     │
│ (指標收集) │                │ (擴展決策) │
└──────────┘                └────┬─────┘
                                 │
                                 ▼
                          ┌──────────────┐
                          │  Deployment  │
                          │ replicas: 3→5│
                          └──────────────┘
```

## Ingress 控制器
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80
```

## Helm 套件管理

```bash
# 安裝 Helm chart
helm install my-release bitnami/wordpress

# 查看部署的 release
helm list

# 升級應用
helm upgrade my-release bitnami/wordpress --set replicas=5

# 回滾
helm rollback my-release 1

# 建立自訂 chart
helm create my-chart
```

## 常見問題處理

| 問題 | 解決方案 |
|------|----------|
| Pod 一直 Pending | 檢查資源配額、節點資源 |
| ImagePullBackOff | 檢查映像檔名稱、registry 認證 |
| OOMKilled | 增加 memory limit |
| CrashLoopBackOff | 檢查應用程式日誌 |
