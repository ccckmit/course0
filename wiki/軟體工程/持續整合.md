# 持續整合

持續整合（Continuous Integration，CI）是 DevOps 實踐中的核心環節，其核心理念是開發團隊成員頻繁地將程式碼整合到共享主線，每次整合都通過自動化建構和測試來驗證，從而盡早發現整合錯誤和 Regression。持續整合不僅僅是一套工具，更是一種開發紀律和文化，它改變了開發團隊的協作方式和軟體品質保障的方式。

## 持續整合的核心概念

### 什麼是持續整合？

傳統開發模式中，開發人員通常在本地工作數天甚至數週，然後將所有程式碼合併到主分支。這種做法存在嚴重問題：

- **整合地獄**：大量程式碼合併時，衝突和錯誤難以解決
- **發現問題延遲**：錯誤直到整合後才被發現，修復成本高昂
- **信心不足**：開發人員不確定自己的程式碼是否正確

持續整合解決了這些問題：
- **頻繁整合**：開發人員每天多次將程式碼推送至共享儲存庫
- **自動化驗證**：每次推送自動觸發建構和測試
- **快速反饋**：盡早發現問題，縮短反饋週期

### 持續整合的工作原理

```
┌─────────────────────────────────────────────────────────────┐
│                    持續整合工作流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  開發者 (Developer)                                          │
│       │                                                     │
│       │  1. 從共享儲存庫 pull 最新程式碼                     │
│       ▼                                                     │
│  本地開發                                                    │
│       │                                                     │
│       │  2. 在功能分支(feature branch)開發                  │
│       ▼                                                     │
│       │  3. 提交並 push 到遠端儲存庫                        │
│       ▼                                                     │
│  Git Server (GitHub/GitLab)                                  │
│       │                                                     │
│       │  4. 觸發 Webhook 通知 CI 伺服器                    │
│       ▼                                                     │
│  CI 伺服器 (Jenkins/GitHub Actions)                         │
│       │                                                     │
│       │  5. 檢出程式碼                                      │
│       │  6. 安裝依賴                                        │
│       │  7. 執行建構                                        │
│       │  8. 執行測試                                        │
│       │  9. 生成報告                                        │
│       ▼                                                     │
│  10. 回饋給開發者                                            │
│       │                                                     │
│       │  - 建構成功：可合併 PR                             │
│       │  - 建構失敗：需要修復                              │
│       ▼                                                     │
│  開發者收到通知 (Email/Slack/Desktop)                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## CI 伺服器的核心功能

### 1. 自動化建構

建構過程通常包括：

```bash
# 典型的建構腳本
#!/bin/bash

# 1. 安裝專案依賴
npm ci              # Node.js
pip install -r requirements.txt  # Python
mvn install         # Java
cargo build         # Rust

# 2. 程式碼品質檢查
npm run lint        # 程式碼風格檢查
npm run audit       # 安全性檢查

# 3. 編譯/打包
npm run build       # TypeScript/JavaScript
mvn package         # Java
go build            # Go

# 4. 產生構件 (Artifacts)
# 產生可部署的 Docker 映像、jar 檔、壓縮包等
```

### 2. 自動化測試

CI 伺服器會執行多層次測試：

```yaml
# 測試配置範例
test:
  - name: Unit Tests
    command: npm run test:unit
    coverage: 80%
    
  - name: Integration Tests
    command: npm run test:integration
    
  - name: E2E Tests
    command: npm run test:e2e
    
  - name: Security Scan
    command: npm audit --audit-level=high
```

### 3. 測試報告生成

```javascript
// Jest 配置生成 Cobertura 格式的覆蓋率報告
// jest.config.js
module.exports = {
  collectCoverage: true,
  coverageReporters: ['text', 'lcov', 'cobertura'],
  coverageThreshold: {
    global: {
      branches: 80,
      functions: 80,
      lines: 80,
      statements: 80
    }
  }
};
```

### 4. 閘道（Quality Gates）

CI 可以設置多種閘道，確保程式碼品質：

```yaml
# CI 閘道配置
quality_gates:
  - name: Code Coverage
    threshold: 80%
    fail: true
    
  - name: Code Style
    tool: eslint
    max_issues: 0
    fail: true
    
  - name: Security Vulnerabilities
    tool: snyk
    severity: high
    fail: true
    
  - name: Test Pass Rate
    threshold: 100%
    fail: true
```

## GitHub Actions 深度詳解

GitHub Actions 是 GitHub 內建的 CI/CD 功能，讓開發者可以在 GitHub 倉庫中創建自動化的工作流。

### 基本概念

- **Workflow（工作流）**：自動化流程的頂層容器
- **Job（工作）**：在工作流中執行的任務組
- **Step（步驟）**：Job 中的單一動作
- **Action（動作）**：可重用的自動化單元

### 完整範例

```yaml
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  NODE_VERSION: '18'
  REGISTRY: ghcr.io

jobs:
  lint:
    name: Lint Code
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4
        
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'
          
      - name: Install Dependencies
        run: npm ci
        
      - name: Run ESLint
        run: npm run lint
        
      - name: Run Prettier Check
        run: npm run format:check

  test:
    name: Run Tests
    runs-on: ubuntu-latest
    needs: lint
    
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
          POSTGRES_DB: test
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432
      
      redis:
        image: redis:7
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 6379:6379
    
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4
        
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'
          
      - name: Install Dependencies
        run: npm ci
        
      - name: Run Unit Tests
        run: npm run test:unit -- --coverage
        
      - name: Run Integration Tests
        run: npm run test:integration
        env:
          DATABASE_URL: postgres://test:test@localhost:5432/test
          REDIS_URL: redis://localhost:6379
          
      - name: Upload Coverage
        uses: codecov/codecov-action@v3
        with:
          token: ${{ secrets.CODECOV_TOKEN }}

  build:
    name: Build Application
    runs-on: ubuntu-latest
    needs: test
    
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4
        
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'
          
      - name: Install Dependencies
        run: npm ci
        
      - name: Build Application
        run: npm run build
        
      - name: Upload Build Artifact
        uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: dist/
          retention-days: 7

  build-docker:
    name: Build Docker Image
    runs-on: ubuntu-latest
    needs: build
    
    permissions:
      contents: read
      packages: write
    
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4
        
      - name: Download Build Artifact
        uses: actions/download-artifact@v4
        with:
          name: build-output
          path: .
          
      - name: Login to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
          
      - name: Build and Push Docker Image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ github.repository }}:${{ github.sha }}
            ${{ env.REGISTRY }}/${{ github.repository }}:latest
          cache-from: type=registry,ref=${{ env.REGISTRY }}/${{ github.repository }}:latest
          cache-to: type=inline

  security-scan:
    name: Security Scan
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4
        
      - name: Run Trivy Scanner
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          scan-ref: '.'
          format: 'sarif'
          output: 'trivy-results.sarif'
          
      - name: Upload Trivy Results
        uses: github/codeql-action/upload-sarif@v2
        with:
          sarif_file: 'trivy-results.sarif'
```

## Jenkins 深度詳解

Jenkins 是最流行的開源 CI/CD 伺服器，擁有豐富的插件生態。

### Jenkinsfile 範例

```groovy
// Jenkinsfile (Declarative Pipeline)
pipeline {
    agent any
    
    environment {
        DOCKER_IMAGE = 'myapp'
        REGISTRY = 'docker.io'
    }
    
    options {
        timeout(time: 30, unit: 'MINUTES')
        disableConcurrentBuilds()
        buildDiscarder(logRotator(numToKeepStr: '10'))
    }
    
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        
        stage('Build') {
            steps {
                script {
                    def dockerImage = docker.build("${env.REGISTRY}/${env.DOCKER_IMAGE}:${env.BUILD_NUMBER}")
                }
            }
        }
        
        stage('Test') {
            parallel {
                stage('Unit Tests') {
                    steps {
                        sh 'npm run test:unit'
                    }
                }
                
                stage('Integration Tests') {
                    steps {
                        sh 'npm run test:integration'
                    }
                }
            }
        }
        
        stage('Security Scan') {
            steps {
                sh 'npm audit --audit-level=high'
                sh 'npx snyk test'
            }
        }
        
        stage('Deploy to Staging') {
            when {
                branch 'develop'
            }
            steps {
                sh 'kubectl apply -f k8s/staging/'
            }
        }
        
        stage('Deploy to Production') {
            when {
                branch 'main'
            }
            steps {
                input message: 'Deploy to Production?', ok: 'Deploy'
                sh 'kubectl apply -f k8s/production/'
            }
        }
    }
    
    post {
        always {
            junit 'test-results/*.xml'
            publishHTML([
                allowMissing: false,
                alwaysLinkToLastBuild: true,
                keepAll: true,
                reportDir: 'coverage',
                reportFiles: 'index.html',
                reportName: 'Coverage Report'
            ])
        }
        
        success {
            slackSend(
                color: 'good',
                message: "Build ${env.BUILD_NUMBER} succeeded!"
            )
        }
        
        failure {
            slackSend(
                color: 'danger',
                message: "Build ${env.BUILD_NUMBER} failed!"
            )
        }
    }
}
```

## CI 最佳實踐

### 1. 小而頻繁的提交

```bash
# 好：小的、專注的提交
git commit -m "Add user validation"
git commit -m "Fix login bug"
git commit -m "Update dependencies"

# 不好：大的、混亂的提交
git commit -m "Various fixes and improvements"
```

### 2. 保持快速的建構時間

- **並行執行測試**：將測試分組並行執行
- **快取依賴**：使用 CI 的快取功能
- **增量建構**：只建構改變的部分
- **容器化**：使用 Docker 確保一致的環境

```yaml
# 快取配置範例
steps:
  - name: Cache npm packages
    uses: actions/cache@v3
    with:
      path: ~/.npm
      key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}
      restore-keys: |
        ${{ runner.os }}-npm-
```

### 3. 建構失敗時立即修復

```python
# 規則：修復失敗的建構是最高優先級
def handle_build_failure():
    while is_build_broken():
        stop_new_work()  # 停止新功能開發
        fix_build()      # 專注修復
        run_tests()      # 驗證修復
        merge_fix()      # 合併修復
```

### 4. 使用功能分支開發

```bash
# 典型的 Git Flow
git checkout -b feature/add-login
# 開發功能...
git push origin feature/add-login
# 創建 PR，CI 自動運行
# Code Review 後合併
```

### 5. 測試環境與開發環境一致

```yaml
# Docker Compose 確保環境一致性
version: '3'
services:
  app:
    build: .
    volumes:
      - .:/app
    depends_on:
      - db
      - redis
      
  db:
    image: postgres:15
    environment:
      POSTGRES_USER: test
      POSTGRES_PASSWORD: test
      
  redis:
    image: redis:7
```

## CI 工具比較

| 工具 | 類型 | 優勢 | 劣勢 |
|------|------|------|------|
| Jenkins | 自託管 | 功能強大、插件豐富、完全免費 | 需要維護伺服器、配置複雜 |
| GitHub Actions | 雲端 | 與 GitHub 深度整合、使用簡單 | 僅限 GitHub |
| GitLab CI | 雲端/自託管 | 一體化 DevOps、CI/CD 免費 | 功能較少 |
| CircleCI | 雲端 | 速度快、易用 | 免費方案有限 |
| Azure Pipelines | 雲端 | 支援多語言、跨平臺 | 與 Azure 綁定 |

## 常見 CI 問題與解決方案

### 1. 建構時間過長

- **解決方案**：使用增量建構、快取依賴、並行執行測試

### 2. 測試不穩定（Flaky Tests）

- **解決方案**：增加重試機制、改善測試隔離、使用合理的等待時間

### 3. 依賴衝突

- **解決方案**：使用鎖定版本（package-lock.json、requirements.txt）

### 4. 環境不一致

- **解決方案**：使用 Docker 容器化所有環境
