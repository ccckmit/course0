# 軟體工程的歷史

軟體工程是將系統化、規範化、可量化的方法應用於軟體開發的學科。從 1968 年 NATO 會議首次提出「軟體工程」一詞開始，這個領域經歷了瀑布模型、敏捷革命、开源運動，至今仍在持續演進。

## 起源：軟體危機（1960-1970 年代）

### 軟體危機的爆發

1968 年，北大西洋公約組織（NATO）在德國加米施召開會議，首次提出了「軟體工程」這個術語。與會者意識到軟體開發正面臨嚴重危機：

- **預算超支**：專案往往超出預算數倍
- **進度延遲**：交付時間不斷推遲
- **品質低下**：錯誤百出，難以維護
- **需求不明**：客戶需求無法清晰定義

IBM 360 作業系統是典型例子：投入了 5000 人年的開發資源，仍問題重重。

### 軟體與硬體的差距

當時的硬體遵循摩爾定律快速進步，但軟體開發，生產力提升緩慢。這形成了著名的「軟體生產力差距」。

## 瀑布模型時代（1970-1980 年代）

### 瀑布模型的提出

1970 年，Winston Royce 發表了《管理大型軟體系統的開發》，提出了瀑布模型：

```
┌─────────────┐
│ 需求分析     │ ← 與客戶討論，收集需求
└──────┬──────┘
       ▼
┌─────────────┐
│ 系統設計     │ ← 高層架構設計
└──────┬──────┘
       ▼
┌─────────────┐
│ 詳細設計     │ ← 模組級設計
└──────┬──────┘
       ▼
┌─────────────┐
│ 程式撰寫     │ ← 編碼
└──────┬──────┘
       ▼
┌─────────────┐
│ 單元測試     │ ← 模組測試
└──────┬──────┘
       ▼
┌─────────────┐
│ 整合測試     │ ← 系統整合
└──────┬──────┘
       ▼
┌─────────────┐
│ 系統測試     │ ← 驗證需求
└──────┬──────┘
       ▼
┌─────────────┐
│ 部署維護     │ ← 交付與維護
└─────────────┘
```

瀑布模型的特點：
- **順序執行**：每個階段完成後才能進入下一階段
- **文件驅動**：每個階段產生詳細文件
- **早期驗證**：在進入開發前明確需求

### 結構化程式設計

1972 年，Edsger Dijkstra 發表了《Go To 語句有害》，推動了結構化程式設計革命。

結構化設計原則：
- 使用順序、選擇、重複結構替代 GOTO
- 模組化設計
- 頂層向下設計（Top-Down Design）

Pascal（1970）和 C（1972）語言支持結構化程式設計。

## 物件導向時代（1980-1990 年代）

### Smalltalk 與物件導向

1972 年，Alan Kay 在 Xerox PARC 發明了 Smalltalk，這是第一個純物件導向語言。

物件導向的核心概念：
- **封裝**：資料和操作封裝在一起
- **繼承**：類別之間的層次關係
- **多型**：同一介面不同實現

物件導向的優勢：
- 提高代碼的可重用性
- 更自然地模擬現實世界
- 更好的資訊隱藏

### C++ 的崛起

1983 年，Bjarne Stroustrup 在貝爾實驗室推出了 C++，結合了 C 的效率和物件導向特性。

C++ 的創新：
- **類**：封裝資料和函數
- **繼承**：單一和多重繼承
- **運算子重載**：自訂運算子行為
- **範本**：泛型程式設計

C++ 成為系統程式和應用軟體的主流語言。

### 設計模式的興起

1994 年，四人組（Gang of Four：Gamma、Helm、Johnson、Vlissides）發表了《設計模式》一書，總結了 23 種常見的物件導向設計模式：

- **Creational**：Singleton、Factory Method、Abstract Factory、Builder、Prototype
- **Structural**：Adapter、Bridge、Composite、Decorator、Facade、Flyweight、Proxy
- **Behavioral**：Chain of Responsibility、Command、Interpreter、Iterator、Mediator、Memento、Observer、State、Strategy、Template Method、Visitor

設計模式成為物件導向軟體設計的標準詞彙。

## 敏捷革命（1990-2000 年代）

### 傳統方法的問題

瀑布模型的問題日益明顯：
- **變更成本高**：需求變更導致大量返工
- **客戶參與少**：客戶只在開始和結束時參與
- **風險暴露晚**：直到項目末期才能看到成果
- **文件過度**：產生大量不實用的文件

### 極限程式設計

1996 年，Kent Beck 在 Chrysler 開發專案中創造了極限程式設計（XP）：

核心實踐：
- **配對程式設計**：兩個程式設計師共用一台電腦
- **測試驅動開發（TDD）**：先寫測試再寫功能
- **重構**：持續改進代碼品質
- **簡單設計**：只做目前需要的功能
- **持續整合**：頻繁合併代碼

### 敏捷宣言

2001 年，17 位軟體專家在猶他州雪鳥度假村會面，發表了《敏捷宣言》：

**價值觀**：
1. 個人和互動優於流程和工具
2. 可工作的軟體優於詳盡的文件
3. 客戶合作優於合約談判
4. 回應變更優於遵循計劃

**原則**：
- 最早的、持續交付有價值的軟體
- 歡迎不斷變更的需求
- 頻繁交付可工作軟體
- 業務人員和開發者每天工作在一起
- 以人為本建立專案
- 面對面交流是最有效的溝通方式
- 可工作軟體是進度的首要度量
- 可持續開發，維持穩定步伐
- 關注技術卓越和良好設計
- 簡單是必要的
- 自組織團隊
- 定期反思並調整行為

### Scrum 方法論

1993 年，Ken Schwaber 和 Jeff Sutherland 共同創建了 Scrum：

Scrum 角色：
- **Product Owner**：負責最大化產品價值
- **Scrum Master**：確保團隊遵循 Scrum
- **開發團隊**：跨功能的自組織團隊

Scrum 事件：
- **Sprint**：1-4 週的迭代
- **Sprint Planning**：規劃迭代內容
- **Daily Standup**：每日同步（15 分鐘）
- **Sprint Review**：展示成果
- **Sprint Retrospective**：反思改進

## 開源運動（1990 年代-至今）

### Linux 的誕生

1991 年，Linus Torvalds 發布了 Linux 核心。Linux 採用 GPL 許可證，允許自由使用、修改和分發。

Linux 的成功因素：
- **開放原始碼**：任何人都可以貢獻
- **Internet 協作**：全球開發者共同維護
- **企業支持**：Red Hat、IBM、Google 等投入資源

如今，Linux 運行在大多數伺服器、超級電腦、Android 手機和雲端基礎設施上。

### 開源軟體生態

1998 年，Netscape 決定開源 Navigator 程式碼，催生了 Mozilla 基金會。同年，Apache HTTP Server 項目成立。

開源軟體的里程碑：
- **Apache**：Web 伺服器
- **MySQL**：關聯式資料庫
- **PHP**：伺服器端腳本
- **Python、Ruby**：腳本語言
- **Git**：版本控制系統
- **Docker**：容器化
- **Kubernetes**：容器編排

### GitHub 與社交編碼

2008 年，GitHub 成立，將 Git 版本控制與社交網路結合。GitHub 讓開源協作變得前所未有地簡單。

如今：
- 開源項目可以在 GitHub 上托管
- Pull Request 機制簡化了代碼審查
- Issue 追蹤取代了傳統的錯誤追蹤系統

## DevOps 時代（2010 年代）

### 傳統 IT 的問題

開發和運維長期分隔：
- **開發團隊**：追求新功能和快速迭代
- **運維團隊**：追求穩定和最小變更
- **衝突**：發布成為瓶頸

### DevOps 的興起

DevOps 打破開孤島，強調開發和運維的協作：

核心實踐：
- **持續整合（CI）**：頻繁合併代碼，自動構建和測試
- **持續部署（CD）**：自動部署到測試和生產環境
- **基礎設施即代碼（IaC）**：用代碼管理基礎設施
- **監控和日誌**：全面了解系統狀態
- **自動化測試**：快速反饋代碼品質

### 雲端原生

雲端運算推動了新的軟體交付模式：

- **容器化**：Docker 打包應用
- **微服務**：將單體應用拆分為小型服務
- **服務網格**：服務間的網路和通訊
- **無伺服器（Serverless）**：函式即服務

## 現代軟體工程（2020 年代）

### AI 輔助開發

大型語言模型開始輔助軟體開發：
- **GitHub Copilot**：AI 程式碼建議
- **ChatGPT**：回答技術問題、解釋代碼
- **AI 重構建議**：自動識別代碼異味

AI 的影響仍處於早期階段，但正在改變開發者的日常工作。

### 平臺工程

平台工程（Platform Engineering）興起，為開發團隊提供內部開發平台（PaaS）：

- 自助式基礎設施
- 標準化的開發工具鏈
- 降低認知負荷

### 開發者體驗

開發者體驗（Developer Experience, DX）成為焦點：
- 快速的本地構建
- 直觀的錯誤訊息
- 良好的文檔

## 中國與台灣的軟體工程

### 台灣的軟體產業

台灣在半導體和電子製造方面領先全球，但在軟體和服務方面仍有提升空間。

重要公司：
- **趨勢科技**：網路安全軟體
- **Garena**：遊戲和電子商務
- **Pinkoi**：設計師電子商務
- **iThome**：雲端服務

台灣的軟體人才在矽谷有重要貢獻。

### 中國的互聯網巨頭

中國的互聯網產業快速發展：
- **阿里巴巴**：電子商務和雲端計算
- **騰訊**：社交和遊戲
- **位元組跳動**：TikTok 和推薦算法
- **百度**：搜尋和 AI

這些公司在雙十一等活動中展示了大規模分散式系統的工程能力。

### 開源的中國力量

中國開發者在開源社區的貢獻不斷增加：
- **阿里巴巴**：Dubbo、RocketMQ、Ant Design
- **位元組跳動**：Volcano、Kratos
- **華為**：openEuler、MindSpore

## 未來趨勢

### 低程式碼/無程式碼

低程式碼平臺（Low-Code/No-Code）讓非技術人員也能構建應用：
- 拖放式介面設計
- 預設邏輯流程
- 快速原型製作

這將 democratize 軟體開發。

### 安全開發一體化

安全左移（Shift Left Security）將安全融入開發流程：
- 靜態應用安全測試（SAST）
- 動態應用安全測試（DAST）
- 軟體組成分析（SCA）

DevSecOps 將安全整合到 DevOps 中。

### 持續學習

軟體工程是快速變化的領域，開發者需要持續學習：
- 新語言和新框架
- 最佳實踐的不斷演化
- 跨領域知識
