# 技術債 (Technical Debt)

## 概述

技術債（Technical Debt）由 Ward Cunningham 提出，是軟體開發中為趕進度而做出的非最佳技術決策，類似金融債務，需要「利息」償還——表現為日後維護困難、開發速度減緩。

## 技術債的成因

### 主動引入
- 為了趕上市場時機接受短期債務
- POC（概念驗證）未及時重構
- 面對壓力選擇捷徑

### 被動產生
- 需求變更導致架構不匹配
- 缺乏技術遠見
- 知識傳遞不足

## 技術債類型

### 程式碼層級
```python
# 範例：重複程式碼
def calculate_discount_user():
    # 計算邏輯
    base_price = 100
    discount = base_price * 0.1
    return discount

def calculate_discount_admin():
    # 幾乎相同的邏輯
    base_price = 100
    discount = base_price * 0.2  # 只有這裡不同
    return discount
```

### 設計層級
- 緊密耦合的模組
- 缺乏抽象層
- 不符合未來擴展的架構

### 測試層級
- 缺乏單元測試
- 低測試覆蓋率
- 脆弱的整合測試

### 文件層級
- 過時的 API 文件
- 缺少架構文件
- 未更新的部署手冊

## 量化技術債

### 技術債比率 (TD Ratio)
```
TD Ratio = 重構成本 / 開發成本
```

### 元度量 (Meta-metrics)
| 指標 | 理想值 | 警告值 |
|------|--------|--------|
| 圈複雜度 | <10 | >20 |
| 耦合度 | 低 | 高 |
| 測試覆蓋率 | >80% | <60% |
| 程式碼重複率 | <3% | >10% |

## 管理技術債

### 追蹤系統
使用 Jira、Linear 等工具追蹤技術債項目，與功能需求同等優先級。

### 償還策略

#### 1. 童子軍規則
> "每次離開時讓程式碼比來時更乾淨"
```python
# 每次修改時
# 1. 重構你碰到的壞味道
# 2. 新增測試覆蓋
# 3. 確保程式碼風格一致
```

#### 2. 20% 時間規則
每 sprint 預留 20% 時間處理技術債。

#### 3. 重構專案
定期規劃專門的重構迭代。

### 重構順序
1. **高影響、高頻使用**：優先重構使用頻繁的核心模組
2. **風險可控**：從低風險區域開始
3. **有測試覆蓋**：確保有足夠測試後再重構

## 技術債管理不當的後果

| 階段 | 症狀 |
|------|------|
| 初期 | 新功能開發變慢 |
| 中期 | Bug 數量增加 |
| 晚期 | 系統不穩定，部署困難 |
