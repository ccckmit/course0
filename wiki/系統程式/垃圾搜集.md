# 垃圾搜集

垃圾搜集（Garbage Collection, GC）是自動記憶體管理的一種形式，由垃圾搜集器自動識別並回收程式不再使用的記憶體。1959 年，John McCarthy 為 LISP 語言首次實現了垃圾搜集，此後這項技術成為許多高階程式語言的核心組成部分。

## 記憶體管理的基本問題

手動記憶體管理長期以來困擾著程式設計師。在 C/C++ 等語言中，程式設計師需要明確分配記憶體（malloc/new）並在不使用時釋放（free/delete）。這一過程容易產生兩類錯誤：

**記憶體洩漏（Memory Leak）**：分配了記憶體但忘記釋放，導致記憶體持續佔用：

```c
// C 語言 - 手動記憶體管理
void process_data() {
    char *buffer = malloc(1024);
    // ... 使用 buffer
    // 忘記 free(buffer) - 記憶體洩漏！
}

int main() {
    while (1) {
        process_data();  // 每次迴圈洩漏 1KB
    }
}
```

**使用已釋放記憶體（Use-After-Free）**：訪問已被釋放的記憶體，導致未定義行為或安全漏洞：

```c
void dangerous() {
    char *data = malloc(100);
    free(data);
    strcpy(data, "corrupted");  // 使用已釋放記憶體！
}
```

垃圾搜集器的核心目標是自動解決這兩個問題，讓程式設計師專注於業務邏輯而非記憶體管理。

## 追蹤式垃圾搜集

追蹤式垃圾搜集（Tracing GC）是目前最主流的垃圾搜集策略。其基本思想是：從根集合（roots）出發，遍歷所有可達（reachable）物件，回收不可達物件所佔用的記憶體。

### 根集合

根集合包括：
- 全域變數（全域物件）
- 棧框架中的局部變數和參數（活躍函數的本地變數）
- 暫存器內容
- 程式碼中的常值（字串等）

### 標記-清除（Mark-and-Sweep）

標記-清除演算法是最基礎的追蹤式 GC，分兩階段工作：

**標記階段（Mark Phase）**：
```
function mark(root):
    for each object in root:
        mark(object)

function mark(obj):
    if obj.marked == false:
        obj.marked = true
        for each reference ref in obj:
            mark(ref)
```

**清除階段（Sweep Phase）**：
```
function sweep():
    for each object in heap:
        if object.marked == false:
            free(object)
        else:
            object.marked = false  // 重置標記
```

標記-清除的優點是實現簡單，缺點是會產生記憶體碎片，且清除階段需要暫停程式執行（Stop-The-World, STW）。

### 標記-壓縮（Mark-Compact）

為了解決記憶體碎片問題，標記-壓縮演算法在標記後將存活物件移動到連續記憶體區域：

```
┌─────────────────────────────────┐
│ 使用中 │ 碎片  │ 使用中 │ 碎片  │  標記-壓縮前
└─────────────────────────────────┘
                ↓
┌─────────────────────────────────┐
│ 使用中 │ 使用中 │      (壓縮後)  │
└─────────────────────────────────┘
```

這消除了碎片但增加了複製開銷，且需要更新所有物件的參照。

### 複製式垃圾搜集（Copying GC）

複製式 GC 將堆記憶體分為兩半（ semispace），每次只使用一半：

```
┌──────────┬──────────┐
│ From Space │ To Space │
└──────────┴──────────┘
     ↑           ↑
   分配      存放存活物件
```

優點：分配簡單快速（指標遞增），無碎片問題。缺點：空間利用率只有 50%。

 Cheney 的複製式 GC 算法利用廣度優先遍歷，有效減少了標記-清除的 STW 時間。

## 引用計數

引用計數（Reference Counting）是最直觀的記憶體管理方式：每個物件維護一個計數器，記錄有多少參照指向它：

```python
# Python 中的引用計數
import sys

a = [1, 2, 3]  # refcount = 1
b = a          # refcount = 2
c = [a, a]     # refcount = 4（a 被引用兩次）
del a           # refcount = 3
del b           # refcount = 2
del c           # refcount = 0，物件被釋放
```

### 引用計數的優點

- **即時回收**：一旦計數歸零，立即釋放記憶體
- **增量執行**：無需 STW，記憶體管理分散在程式執行中
- **可預測性**：記憶體使用可控，無不可預知的 GC 暫停

### 引用計數的缺點

- **無法處理循環參照**：

```python
class Node:
    def __init__(self):
        self.next = None

a = Node()
b = Node()
a.next = b  # a 引用 b
b.next = a  # b 引用 a
# 即使刪除 a, b，循環引用導致計數不為零
del a
del b  # 記憶體洩漏！
```

- **計數器更新開銷**：每次參照變化都要更新計數
- **儲存開銷**：每個物件需要額外空間存計數

Python 同時使用引用計數（立即回收）和標記-清除（處理循環），互補不足。

## 分代垃圾搜集

分代假設（Generational Hypothesis）提出：大多數物件都是「朝生夕死」的，只有少數物件存活時間很長。基於這一假設，分代 GC 將物件按年齡分組，針對不同代採用不同策略。

### 基本結構

```
┌─────────────────────────────────────────┐
│           年輕代 (Young Generation)       │
│  ┌─────────┐ ┌─────────┐ ┌─────────────┐│
│  │ Eden    │ │ Survivor│ │ Survivor 1  ││
│  │         │ │ Space 0 │ │ (To)        ││
│  └─────────┘ └─────────┘ └─────────────┘│
│                                         │
│  ┌─────────────────────────────────────┐│
│  │       老年代 (Old Generation)         ││
│  └─────────────────────────────────────┘│
└─────────────────────────────────────────┘
```

### Minor GC vs Major GC

- **Minor GC**：發生在年輕代，頻率高但回收快（大部分物件在此死亡）
- **Major GC/Full GC**：發生在老年代，頻率低但暫停時間長

### 跨代引用問題

年輕代物件可能引用老年代物件。為避免掃描整個老年代，引入「記憶卡表」（Card Table）追蹤跨代引用：

```
記憶卡表: [DIRTY] [CLEAN] [DIRTY] [CLEAN]
           區塊1   區塊2   區塊3   區塊4
```

當老年代物件修改對年輕代的引用時，對應記憶卡被標記為 DIRTY，GC 時只需掃描 DIRTY 記憶卡。

## 即時（JIT）編譯與 GC

現代 JVM（如 HotSpot）將 GC 與 JIT 編譯深度整合：

### 自適應優化

JVM 監控程式執行，動態調整優化策略：
- 熱點偵測：多次執行的方法被 JIT 編譯
- 反最佳化：若假設失敗，回退到直譯執行

### GC 調優參數

```bash
# 選擇 GC 演算法
-XX:+UseG1GC      # G1 收集器
-XX:+UseZGC       # ZGC（低延遲）
-XX:+UseShenandoahGC  # Shenandoah

# 記憶體配置
-Xms2g -Xmx2g      # 初始/最大堆記憶體
-XX:NewRatio=2     # 年輕代/老年代比例
```

## 各語言的 GC 策略

| 語言 | GC 策略 | 特點 |
|------|---------|------|
| Java | 分代 GC（G1、ZGC、Shenandoah） | 可配置的延遲/吞吐量權衡 |
| C# | 分代 GC + 參照計數（Span<T>） | 增量式收集 |
| Go | 並行標記-清除 | 低延遲、MUTator 並行 |
| Python | 參照計數 + 備用 GC | 即時回收大部分 |
| JavaScript | 標記-清除 + 增量標記 | V8 引擎 |
| Rust | 無 GC（所有權系統） | 編譯時靜態分析 |
| Swift | 自動參照計數（ARC） | 無循環引用檢測 |

## 記憶體安全與 GC

垃圾搜集是實現記憶體安全的重要手段之一：

- **無-use-after-free**：已釋放記憶體不可能被使用
- **無緩衝區溢位**：動態分配的陣列有邊界檢查
- **無雙重釋放**：物件只會被釋放一次

對比 Rust 的靜態記憶體安全：
- GC：執行時安全，Runtime 開銷
- Rust：編譯時安全，無 Runtime 開銷
