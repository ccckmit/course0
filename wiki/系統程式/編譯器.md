# 編譯器

編譯器是電腦科學中最重要的系統軟體之一，其核心使命是將人類可讀的高階程式語言（如 C、C++、Java、Python）翻譯成機器可直接執行的機器碼（或目標碼）。這個翻譯過程不僅僅是簡單的字元轉換，而是一個複雜的多階段系統工程，涉及語法分析、語意理解、優化轉換和碼生成等多個層次的處理。

## 編譯器的歷史演進

編譯器的發展歷程反映了電腦科學的進步。1950年代，早期的編譯器（如 FORTRAN 編譯器）開創了自動翻譯高階語言的先河，此後經歷了數十年的演進：

- **第一代編譯器（1950-1970）**：專注於基本翻譯功能，主要支援 FORTRAN、LISP、COBOL 等早期語言
- **第二代編譯器（1970-1990）**：引入最佳化技術，出現了 GCC、LLVM 等代表性編譯器框架
- **第三代編譯器（1990-至今）**：強調跨平臺編譯、即時編譯（JIT）、多語言前端支援

## 編譯器的核心組成

### 1. 詞法分析（Lexical Analysis）

詞法分析是編譯過程的第一個階段，其任務是將原始程式碼（字元流）轉換為 Token 序列。Token 是具有語意價值的最小語法單位，包括：

- **關鍵字**（Keywords）：如 `if`、`while`、`return` 等語言保留字
- **識別符**（Identifiers）：變數名、函數名、類別名等
- **常值**（Literals）：數值、字串、布林值等
- **運算子**（Operators）：`+`、`-`、`*`、`/`、`=` 等
- **分隔符**（Delimiters）：`(`、`)`、`{`、`}`、`;`、`,` 等

詞法分析器（Lexer）通常使用正則表達式或有限狀態機（Finite State Machine）來識別這些 Token。例如，一個辨識整數常值的有限狀態機可能具有以下狀態：初始狀態 → 讀取數字 → 繼續讀取數字 → 完成。

典型工具包括 Lex、Flex 等。

### 2. 語法分析（Syntax Analysis）

語法分析階段將 Token 序列轉換為抽象語法樹（Abstract Syntax Tree, AST）。這個階段驗證程式碼是否符合語言的文法規則，並建立程式的結構化表示。

語法分析的核心概念包括：

- **文法（Grammar）**：描述語言語法的形式化系統，通常使用 BNF（Backus-Naur Form）或 EBNF 表示
- **解析器（Parser）**：根據文法規則解析 Token 序列的程式
- **抽象語法樹（AST）**：程式的樹狀結構表示，每個節點代表一個語法結構

常見的語法分析方法包括：

- **自頂向下解析**：從根節點開始向下推導，如遞迴下降解析、LL(1) 解析
- **自底向上解析**：從葉節點開始向上歸約，如 LR 解析、LALR 解析

語法分析器生成的 AST 保留了程式的語法結構，但去除了多餘的語法細節（如括號、分號等），為後續的語意分析提供更清晰的表示。

### 3. 語意分析（Semantic Analysis）

語意分析是編譯過程中確保程式意義正確的關鍵階段。它在語法正確的基礎上，進一步檢查程式的語意正確性，包括：

- **型態檢查（Type Checking）**：確保運算元的型態相容，例如不能將字串與整數相加
- **作用域檢查（Scope Checking）**：確保變數在使用前已經宣告，識別符的引用是正確的
- **唯一性檢查**：確保沒有重複宣告（如同一作用域內的兩個同名變數）
- **控制流檢查**：確保所有程式路徑都有返回值（如函數必須返回指定型態的值）

語意分析通常使用符號表（Symbol Table）來追蹤識別符的宣告與引用，並利用屬性文法（Attribute Grammar）來計算和傳遞語意資訊。

### 4. 中間表示式（Intermediate Representation, IR）

中間表示式是編譯器內部使用的程式表示形式，它既不是原始的高階語言，也不是最終的機器碼，而是一種介於兩者之間的表示形式。IR 的設計目標是：

- **平臺無關性**：與目標硬體無關，可支援多種來源語言和目標平臺
- **易於優化**：提供豐富的優化機會，便於進行各種程式轉換
- **易於分析**：支持靜態分析和程式碼檢測

常見的 IR 形式包括：

- **三地址碼（Three-Address Code, TAC）**：每條指令最多三個運算元，如 `t1 = a + b`
- **靜態單一賦值（Static Single Assignment, SSA）**：每個變數只被賦值一次，是現代編譯器優化的基礎
- **控制流圖（Control Flow Graph, CFG）**：以圖的形式表示程式的控制流結構

LLVM 的 LLVM IR 是最廣泛使用的 IR 之一，它是一種強類型的低級表示式，支持豐富的元數據和調試信息。

### 5. 程式碼最佳化（Optimization）

程式碼最佳化是編譯器中提升程式效能的關鍵階段，它在保持程式語意的基礎上，盡可能提升程式的執行效率、減少資源消耗。最佳化可以在不同層次進行：

#### 5.1 本地最佳化（Local Optimizations）

在單一基本區塊（Basic Block）內進行的最佳化：

- **常數折疊（Constant Folding）**：編譯時計算常數表達式，如 `3 + 5` → `8`
- **代數簡化（Algebraic Simplification）**：簡化代數表達式，如 `x + 0` → `x`
- **消除公共子表達式（Common Subexpression Elimination）**：避免重複計算相同的表達式
- **死碼消除（Dead Code Elimination）**：移除永遠不會執行的程式碼

#### 5.2 全域最佳化（Global Optimizations）

跨越多個基本區塊的最佳化：

- **活躍變數分析（Live Variable Analysis）**：確定哪些變數在後續仍會被使用
- **資料流分析（Data Flow Analysis）**：追蹤資料在程式中的流動
- **迴圈最佳化（Loop Optimizations）**：包括迴圈展開、迴圈合併、迴圈不變代碼移動等

#### 5.3 目標機器最佳化

針對特定硬體平臺的最佳化：

- **指令排程（Instruction Scheduling）**：優化指令執行順序以提升管線效率
- **暫存器分配（Register Allocation）**：有效分配有限數量的暫存器
- **指令選擇（Instruction Selection）**：選擇最適合目標硬體的指令

### 6. 目的碼生成（Code Generation）

目的碼生成是編譯的最後階段，它將最佳化後的中間表示式轉換為目標機器的機器碼。這一階段需要考慮：

- **指令選擇**：將 IR 操作映射到目標機器的指令
- **暫存器分配**：決定何時使用暫存器、何時使用記憶體
- **呼叫慣例（Calling Convention）**：處理函數參數傳遞、返回值、堆疊框架等
- **ABI 合規**：確保生成的程式碼符合應用二進制介面規範

目標碼可以是：

- **組譯碼（Assembly Code）**：人類可讀的機器碼表示
- **目標檔（Object File）**：二進制格式的機器碼，包含符號表、重定位資訊等
- **可執行檔（Executable）**：完整的可執行程式

## 現代編譯器框架

### LLVM（Low Level Virtual Machine）

LLVM 是目前最具影響力的開源編譯器基礎設施，由 Chris Lattner 於 2000 年在伊利諾大學開發。LLVM 的核心設計理念是提供一個模組化、可重用的編譯器骨架：

- **Clang**：C/C++/Objective-C 前端，提供快速的編譯和精確的診斷資訊
- **LLVM IR**：統一的中間表示式，支援多種語言和目標平臺
- **_TARGET**：支援 x86、ARM、RISC-V、GPU 等多種目標架構
- **MLIR**：新的多層次中醫表示式，支持硬體加速器編譯

### GCC（GNU Compiler Collection）

GCC 是 GNU 計畫的核心組件，支援 C、C++、Fortran、Java、Go 等多種語言，是 Linux 系統的主要編譯器。GCC 採用傳統的單一語言前端設計，但通過後端的重複使用實現多語言支持。

## 編譯器的應用場景

1. **系統軟體開發**：作業系統、驅動程式、嵌入式系統
2. **應用軟體開發**：桌面應用、行動應用、遊戲
3. **效能關鍵應用**：科學計算、資料處理、機器學習
4. **編譯器自身開發**：元編譯器、語言實現

## 編譯器與其他領域的關係

- **形式文法**：提供語言語法的理論基礎
- **型態系統**：語意分析的重要組成
- **作業系統**：目標碼的執行環境
- **電腦架構**：指令集設計影響編譯器後端
- **軟體工程**：編譯器的可靠性和錯誤訊息品質影響開發效率
