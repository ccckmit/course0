# 排程

排程（Scheduling）是作業系統的核心功能之一，決定在任意時刻哪個程序應該獲得 CPU 執行權。一個好的排程演算法能夠最大化 CPU 利用率、最小化回應時間、同時確保公平性與即時性。

## 排程目標

排程器需要平衡多個往往相互衝突的目標：

```
┌─────────────────────────────────────────────────────────────────┐
│                     排程目標                                │
├─────────────────────────────────────────────────────────────────┤
│                                                              │
│  • CPU 利用率         CPU 應該盡可能忙碌                       │
│  • 吞吐量            單位時間完成的工作量                      │
│  • 周轉時間          從提交到完成的總時間                      │
│  • 等待時間          在就緒佇列中等待的時間                   │
│  • 響應時間          從提交到首次響應的時間                    │
│  • 公平性           每個程序獲得公平的 CPU 時間               │
│  • 即時性            嚴格的時間約束                          │
│                                                              │
└─────────────────────────────────────────────────────────────────┘
```

### 效能指標

| 指標 | 定義 | 理想值 |
|------|------|--------|
| CPU 利用率 | CPU 工作時間 / 總時間 | 100% |
| 吞吐量 | 每單位時間完成的程序 | 高 |
| 周轉時間 | 完成時間 - 到達時間 | 低 |
| 等待時間 | 就緒佇列中等待的總時間 | 低 |
| 響應時間 | 首次響應時間 - 到達時間 | 低 |

## 排程演算法

### 非搶佔式排程

程序一旦獲得 CPU，就一直執行直到阻塞或完成：

```c
// FCFS (First-Come-First-Served)
// 按到達順序服務
// 優點：簡單、開銷小
// 缺點：響應時間不均勻

// SJF (Shortest Job First)
// 優先服務短工作
// 優點：最優的平均等待時間
// 缺點：長工作可能餓死
// 需要知道工作長度

// 優先級排程
// 高優先級程序優先
// 可能導致低優先級餓死
```

### 搶佔式排程

更高優先級的程序可以搶佔正在執行的程序：

```c
// 時間片輪轉 (Round Robin)
// 每個程序分配時間片
// 時間片用完則移到佇列尾
// 優點：公平、響應快
// 缺點：上下文切換開銷

// 多級佇列
// 多個就緒佇列，不同優先級
// 佇列間固定優先級
// 佇列內 FCFS 或 RR

// 多級回饋佇列 (MLFQ)
// 允許程序在佇列間移動
// 使用更多 CPU 的程序降級
// 等待久的程序升級
// 兼顧公平與效率
```

### 演算法比較

| 演算法 | CPU 利用 | 吞吐量 | 周轉 | 等待 | 響應 | 公平 |
|--------|---------|--------|------|------|------|------|
| FCFS | 高 | 低 | 高 | 高 | 低 | 好 |
| SJF | 中 | 高 | 低 | 低 | 高 | 差 |
| RR | 高 | 高 | 中 | 低 | 好 | 好 |
| MLFQ | 高 | 高 | 低 | 低 | 好 | 好 |
| 優先級 | 不一定 | 不一定 | 不一定 | 不一定 | 不一定 | 差 |

## 多級回饋佇列

Linux CFS 之前廣泛使用的經典演算法：

```
┌───────────���─────────────────────────────────────────────────────┐
│                    多級回饋佇列                              │
├─────────────────────────────────────────────────────────────────┤
│  優先級高  │  佇列 Q1   │  時間片 10ms │  RR                  │
│              ├────────────┼─────────────┤                      │
│              │  佇列 Q2   │  時間片 20ms │  RR                  │
│              ├────────────┼─────────────┤                      │
│              │  佇列 Q3   │  時間片 40ms │  RR                  │
│              ├────────────┼─────────────┤                      │
│              │  佇列 Q4   │  時間片 80ms │  FCFS                │
└─────────────────────────────────────────────────────────────────┘

規則：
1. 新程序進入最高佇列
2. 用完時間片降級
3. 未用完時間片阻塞則保留級別
4. 高優先級佇列為空才運行低優先級
```

## Linux CFS 排程

CFS（Completely Fair Scheduler）是 Linux 2.6.23 引入的完全公平排程器：

```c
// 核心概念：
// - 虛擬執行時間 (vruntime)
// - 每個程序的理想執行時間
// - 紅黑樹管理就緒程序
// - 最左葉子節點獲得 CPU

// 設計目標：
// - 公平：每個程序獲得相等 CPU 時間
// - 低開銷：O(log n) 的紅黑樹
// - 低延遲：互動任務快速回應
// - 高吞吐量：背景任務充分運行

// 權重 (weight)
// nice 值影響權重
// nice -19 → weight 88761
// nice 0  → weight 1024
// nice +19 → weight 15
```

### CFS 程式碼結構

```c
struct sched_entity {
    u64             vruntime;      // 虛擬執行時間
    u64             sum_exec_runtime;
    u64             prev_sum_exec_runtime;
    u64             last_wakeee;
    unsigned long       load;
    struct rb_node      run_node;
    struct list_head   group_node;
    unsigned int       on_rq;
    u64             exec_start;
};

struct cfs_rq {
    struct rb_root_cudo   tasks_timeline;
    struct rb_node *rb_leftmost;
    struct sched_entity *curr;
    u64             tglock;
    unsigned long   nr_running;
    unsigned long   h_nr_running;
};
```

## 即時排程

即時系統要求嚴格的時間約束：

```
┌─────────────────────────────────────────────────────────────────┐
│                      即時系統                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                              │
│  硬即時 (Hard Real-time)                                    │
│  - 必須嚴格在截止時間前完成                                  │
│  - 超過截止時間 = 系統失敗                                   │
│  - 例：飛行控制、核電廠                                     │
│                                                              │
│  軟即時 (Soft Real-time)                                    │
│  - 偶爾超過截止時間可接受                                   │
│  - 例：視訊播放、網路電話                                   │
│                                                              │
└─────────────────────────────────────────────────────────────────┘
```

### Rate Monotonic Scheduling (RMS)

```c
// 假設：
// - 任務是週期性的
// - 任務必須在週期內完成
// - 任務是獨立的
// - 上下文切換時間可忽略

// 規則：
// - 週期越短，優先級越高

// 可行性測試：
// // n = 任務數
// // Pi = 週期, Ci = 執行時間
// // U = Σ(Ci/Pi) ≤ n(2^(1/n) - 1)
// // 實際可到 100% 利用率
```

### Earliest Deadline First (EDF)

```c
// 規則：
// - 截止時間越早，優先級越高
// - 動態優先級

// 可行性測試：
// // 對任何時刻 t
// // Σ(Ci(Pi ≤ t) / Pi) ≤ 1
// // 可以達到 100% 利用率

// 優點：利用率更高
// 缺點：需要動態計算，開銷大
```

## 上下文切換

程序切換涉及的開銷：

```c
// 上下文切換步驟：
// 1. 保存目前程序暫存器
// 2. 更新程序控制區塊
// 3. 移動程序到合適佇列
// 4. 選擇新程序
// 5. 恢復新程序暫存器
// 6. 切換到新程序的位址空間

// 快取失效：
// - L1/L2/L3 快取失效
// - TLB 刷新
// - 分支預測失效
// - 典型的上下文切換 1-10μs
// - 快取失效可能 10-100μs
```

```c
// 減少上下文切換：
// - 合併多個小任務
// - 使用合適的鎖
// - 減少 CPU 跨越
// - 避免過度執行緒化
```

## 排程器類型

| 類型 | 說明 | 範例 |
|------|------|------|
| 長期排程 | 決定進入系統的程序數 | 批次處理系統 |
| 短期排程 | 決定就緒程序誰執行 | 所有系統 |
| 中期排程 | 臨時移出程序到交換區 | Linux, Windows |
