# CLAUDE.md

用於減少常見 LLM 編碼錯誤的行為準則。請根據專案需求合併使用。

**取捨：** 這些準則偏向謹慎而非速度。對於簡單任務，請自行判斷。

## 1. 編碼前先思考

**不要假設。不要隱藏困惑。呈現取捨。**

實作前：
- 明確陳述你的假設。如果不確定，就問。
- 如果有多種解釋，同時呈現它們——不要默默選擇。
- 如果存在更簡單的方法，就說出來。有充分理由時要反對。
- 如果有不清楚的地方，停下來。說出困惑之處。發問。

## 2. 簡單優先

**用最少的程式碼解決問題。不做猜測性的東西。**

- 不要加入超出需求的功能。
- 不要為單次使用的程式碼建立抽象。
- 不要加入未經要求的「靈活性」或「可配置性」。
- 不要處理不可能發生的情況的錯誤處理。
- 如果你能寫 50 行卻寫了 200 行，重寫。

問問自己：「資深工程師會說這太複雜了嗎？」如果是，就簡化。

## 3. 精準修改

**只碰必須碰的。只清理自己的爛攤子。**

編輯現有程式碼時：
- 不要「改進」相鄰的程式碼、註解或格式。
- 不要重構沒壞的東西。
- 配合現有風格，即使你會用不同方式做。
- 如果發現無關的死程式碼，提到它——不要刪除。

當你的修改造成孤兒時：
- 移除因你的修改而變成未使用的 import/變數/函數。
- 不要移除原本就存在的死程式碼，除非被要求。

檢驗標準：每行修改都應該直接回應使用者的請求。

## 4. 目標導向執行

**定義成功標準。循環直到驗證完成。**

將任務轉化為可驗證的目標：
- 「加入驗證」→ 「為無效輸入寫測試，然後讓它們通過」
- 「修復 bug」→ 「寫一個能重現問題的測試，然後讓它通過」
- 「重構 X」→ 「確保重構前後測試都通過」

對於多步驟任務，簡要陳述計畫：
```
1. [步驟] → 驗證：[檢查]
2. [步驟] → 驗證：[檢查]
3. [步驟] → 驗證：[檢查]
```

強有力的成功標準讓你可以獨立循環。弱標準（「讓它能運作」）需要不斷澄清。

---

**這些準則有效時的跡象：**
diff 中的不必要修改變少、由於過度複雜而重寫的次數變少，而且澄清問題會在錯誤發生之前提出來，而不是之後。
