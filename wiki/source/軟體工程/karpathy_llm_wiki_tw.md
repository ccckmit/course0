# LLM Wiki

一個使用 LLM 建立個人知識庫的模式。

這是一個想法檔案，設計成可以複製貼上到你自己的 LLM Agent（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目標是傳達高層概念，但你的 agent 會與你合作建立具體細節。

## 核心概念

大多數人與 LLM 及文件互動的方式就像 RAG：你上傳一批檔案，LLM 在查詢時檢索相關片段並產生答案。這能運作，但 LLM 是每次問題都從頭重新發現知識。沒有累積。問一個需要綜合五份文件的微妙問題，LLM 每次都必須找到並拼湊相關片段。沒有任何建構。NotebookLM、ChatGPT 檔案上傳和大多的 RAG 系統都是這樣運作。

這裡的想法不同。不只是在查詢時從原始文件檢索，LLM **增量建構和維護一個持久的 wiki** —— 一個結構化的、互相連結的 markdown 檔案 collection，坐落於你和原始來源之間。當你加入新來源時，LLM 不只是將它索引以供日後檢索。它會閱讀它、擷取關鍵資訊，並將其整合到現有 wiki —— 更新實體頁面、修訂主題摘要、標記新資料與舊聲明矛盾的地方、加強或挑戰演進中的綜合。知識編譯一次然後*保持最新*，而不是每次查詢都重新推導。

這是關鍵差異：**wiki 是一個持久的、複合的產物。** 交叉引用已經存在。矛盾已經被標記。綜合已經反映了你讀過的一切。wiki 隨著你加入的每個來源和每個問題變得越來越豐富。

你（或很少）自己寫 wiki —— LLM 寫作和維護這一切。你負責來源、探索和提出正確的問題。LLM 做所有繁重的工作——摘要、交叉引用、歸檔和簿記，讓知識庫隨著時間變得真正有用。在實踐中，我讓 LLM agent 開在一邊，Obsidian 開在另一邊。LLM 根據我們的對話進行編輯，我即時瀏覽結果——跟隨連結、查看圖形視圖、閱讀更新的頁面。Obsidian 是 IDE；LLM 是程式設計師；wiki 是程式碼庫。

這可以應用於許多不同的上下文。例如：

- **個人**：追蹤自己的目標、健康、心理、自我改進——歸檔日誌文章、播客筆記，並隨著時間建立一個結構化的自我畫像。
- **研究**：深入研究一個主題數週或數月——閱讀論文、文章、報告，並逐步建立一個包含演進論點的綜合 wiki。
- **閱讀書籍**：隨著閱讀歸檔每個章節，為角色、主題、情節線條及其如何連接建立頁面。到最後你會有一個豐富的伴讀 wiki。想想類似 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 的粉絲 wiki —— 數千個互相連結的頁面涵蓋角色、地點、事件、語言，由志願者社群多年建構。你可以在閱讀時自己建立類似的東西，由 LLM 完成所有交叉引用和維護。
- **商務/團隊**：由 LLM 維護的內部 wiki，由 Slack 討論串、會議記錄、專案文件、客戶電話餵入。可能有人類參與審查更新。wiki 保持最新是因為 LLM 做了團隊中沒人想做的維護工作。
- **競爭分析、尽职调查、旅程規劃、課程筆記、嗜好深入研究** —— 任何你隨著時間累積知識且希望它是有組織的而不是散亂的上下文。

## 架構

有三層：

**原始來源** —— 你策展的來源文件 collection。文章、論文、圖片、資料檔案。這些是不可變的 —— LLM 讀取它們但從不修改它們。這是你的事實來源。

**The wiki** —— 一個 LLM 產生的 markdown 檔案目錄。摘要、實體頁面、概念頁面、比較、概述、綜合。LLM 完全擁有這層。它建立頁面、在新來源到達時更新它們、維護交叉引用並保持一切一致。你閱讀它；LLM 寫作它。

**schema** —— 一個文件（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md）告訴 LLM wiki 是如何組織的、 convention 是什麼，以及在攝取來源、回答問題或維護 wiki 時要遵循什麼工作流程。這是關鍵的配置檔案 —— 這是讓 LLM 成為一個有紀律的 wiki 維護者而不是通用聊天機器人的原因。你和 LLM 隨著時間共同演進這個，因為你弄清楚什麼適合你的領域。

## 操作

**攝取。** 你將新來源放入原始 collection 並告訴 LLM 處理它。一個範例流程：LLM 閱讀來源、與你討論關鍵要點、在 wiki 中寫一個摘要頁面、更新索引、更新 wiki 中相關的實體和概念頁面，並附加一筆記錄到日誌。單一來源可能會觸及 10-15 個 wiki 頁面。個人來說，我偏好一次攝取一個來源並保持參與——我閱讀摘要、檢查更新，並引導 LLM 強調什麼。但你也可以在較少監督的情況下批次攝取多個來源。由你決定開發適合你風格的工作流程並將其記錄在 schema 中以供未來的會議使用。

**查詢。** 你對 wiki 提出問題。LLM 搜尋相關頁面、閱讀它們，並綜合一個帶有引用 的答案。答案可以根據問題採取不同的形式——一個 markdown 頁面、比較表格、投影片 deck（Marp）、圖表（matplotlib）、畫布。重要的洞察：**好的答案可以歸檔回 wiki 成為新頁面。** 你要求的比較、分析、你發現的連接——這些是有價值的，不應該消失在聊天歷史中。這樣你的探索就像攝取的來源一樣在知識庫中複合。

**Lint。** 定期請 LLM 對 wiki 進行健康檢查。尋找：頁面之間的矛盾、被新來源取代的過時聲明、沒有入站連結的孤立頁面、有提到但缺少自己頁面的重要概念、缺失的交叉引用、可以用網路搜尋填補的資料缺口。LLM 善於建議要調查的新問題和要尋找的新來源。這讓 wiki 在成長時保持健康。

## 索引和日誌

兩個特殊檔案幫助 LLM（和你）導航成長中的 wiki。它們服務不同的目的：

**index.md** 是內容導向的。它是 wiki 中一切的目錄——每個頁面列出帶有連結、一行摘要，以及可選的元數據如日期或來源數量。按類別組織（實體、概念、來源等）。LLM 在每次攝取時更新它。回答查詢時，LLM 先閱讀索引以找到相關頁面，然後深入閱讀它們。這在中等規模（約 100 個來源、數百個頁面）時出人意料地有效，避免了對基於嵌入的 RAG 基礎設施的需求。

**log.md** 是時間順序的。這是一個僅附加的記錄，記錄發生了什麼以及何時——攝取、查詢、lint 通過。一個有用的提示：如果每筆記錄以一致的前綴開頭（例如 `## [2026-04-02] ingest | Article Title`），日誌可以用簡單的 unix 工具解析—— `grep "^## \[" log.md | tail -5` 給你最後 5 筆記錄。日誌給你 wiki 演化的時間線，並幫助 LLM 理解最近做了什麼。

## 可選：CLI 工具

在某個時候，你可能想要建構小工具來幫助 LLM 更有效地操作 wiki。對 wiki 頁面的搜尋引擎是最明顯的一個——在小規模時索引檔案就夠了，但隨著 wiki 成長，你需要適當的搜尋。[qmd](https://github.com/tobi/qmd) 是一個好選項：它是一個本地 markdown 檔案搜尋引擎，帶有混合的 BM25/向量搜尋和 LLM 重新排名，全部在設備上。它有 CLI（所以 LLM 可以 shell 出到它）和 MCP 伺服器（所以 LLM 可以將其用作原生工具）。你也可以自己建構更簡單的東西——LLM 可以幫助你在需要時 vibe-code 一個簡單的搜尋腳本。

## 技巧和訣竅

- **Obsidian Web Clipper** 是一個瀏覽器擴充功能，將網路文章轉換為 markdown。對於快速將來源放入你的原始 collection 非常有用。
- **將圖片下載到本���。** 在 Obsidian 設定 → Files and links，將「Attachment folder path」設定為固定目錄（例如 `raw/assets/`）。然後在設定 → Hotkeys 中，搜尋「Download」找到「Download attachments for current file」並將其綁定到快捷鍵（例如 Ctrl+Shift+D）。在剪輯文章後，按下快捷鍵，所有圖片都會下載到本地磁碟。這是可選但有用的——它讓 LLM 可以直接查看和引用圖片，而不是依賴可能失效的 URL。請注意，LLM無法原生一次閱讀帶有內聯圖片的 markdown —— 解決方法是讓 LLM 先閱讀文字，然後單獨查看部分或全部引用的圖片以獲得額外上下文。這有點笨拙但效果足夠。
- **Obsidian 的圖形視圖** 是查看你的 wiki 形狀的最佳方式——什麼連接到什麼、哪些頁面是中心、哪些是孤兒。
- **Marp** 是一個基於 markdown 的投影片 deck 格式。Obsidian 有它的外掛。對於直接從 wiki 內容產生簡報很有用。
- **Dataview** 是一個 Obsidian 外掛，對頁面 frontmatter 執行查詢。如果你的 LLM 為 wiki 頁面加入 YAML frontmatter（標籤、日期、來源計數），Dataview 可以產生動態表格和列表。
- wiki 只是一個 markdown 檔案的 git 儲存庫。你免費獲得版本歷史、分支和協作。

## 為什麼這有效

維護知識庫的繁瑣部分不是閱讀或思考——而是簿記。更新交叉引用、保持摘要最新、標記新資料與舊聲明矛盾的地方、維護數十頁的一致性。人類放棄 wiki 是因為維護負擔成長比價值快。LLM 不會無聊、不會忘記更新交叉引用，並且可以一次觸及 15 個檔案。wiki 保持維護是因為維護成本接近零。

人類的工作是策展來源、引導分析、提出好問題，並思考這一切意味著什麼。LLM 的工作是其他一切。

這個想法在精神上與 Vannevar Bush 的 Memex（1945）相關——一個帶有文件之間關聯軌跡的個人策展知識商店。Bush 的願景比網路更接近這個：私人、積極策展，文件之間的連接與文件本身一樣有價值。他無法解決的部分是誰做維護。LLM 處理那個。

## 註記

這份文件故意是抽象的。它描述想法，而不是具體的實現。確切的目錄結構、schema convention、頁面格式、工具——所有這些都取決於你的領域、你的偏好的你選擇的 LLM。以上提及的一切都是可選和模組化的——選擇有用的，忽略沒有的。例如：你的來源可能只有文字，所以你根本不需要圖片處理。你的 wiki 可能很小，只需要索引檔案，不需要搜尋引擎。你可能不在乎投影片只想看 markdown 頁面。你可能想要完全不同的輸出格式集合。使用這個的正確方式是與你的 LLM agent 分享它並一起合作來實例化一個適合你需要 的版本。這份文件的唯一工作是傳達模式。你的 LLM 可以弄清楚其餘的。
